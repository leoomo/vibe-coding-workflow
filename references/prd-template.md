# PRD 模板

> Step 2 Writer 起草 `prd.md` 用。先把"做什么"钉死，再去想"怎么做"。

## prd.md 格式

```markdown
# PRD: <功能名称>

## 1. 背景与问题
- 要解决什么问题？现状有什么痛点？
- 为什么现在做？

## 2. 目标用户
- 谁会用？角色/权限分别是什么？

## 3. 核心使用场景
- 场景 A：<谁>在<何时>为了<什么>做<操作>，期望<结果>
- 场景 B：<…>

## 4. 功能范围（In Scope）
- <本次要交付的具体功能点>

## 5. 不在本次范围（Out of Scope）
- <明确不做的，避免范围蔓延>

## 6. 边界情况
- 空数据 / 重复提交 / 并发 / 超长输入 / 越权访问 / 网络失败 / 旧数据兼容 …

## 7. 成功标准（可验收）
- <可观测、可验证的判定标准，尽量量化>
- 例：邀请 3 个用户后，3 人均收到邮件且能完成注册

## 8. 复杂逻辑清单（逐项确认 有/无/如何处理）
- [ ] 权限控制
- [ ] 计费 / 配额
- [ ] 通知（邮件/站内/推送）
- [ ] 数据迁移 / 向后兼容
- [ ] 国际化 / 时区
- [ ] 审计日志

## 9. 非功能需求（如有）
- 性能 / 安全 / 可观测性 / 兼容性
```

## 起草要点

- **范围要小而清**：第 4、5 节是防止返工的关键，宁可写细。
- **边界要全**：第 6 节直接影响 Step 3 的测试设计，漏了边界 = 漏了测试。
- **成功标准要可验收**：第 7 节是 Step 6 验收门禁的依据。
- **复杂逻辑要逐项点名**：第 8 节逐项打勾，防止"忘了考虑计费/迁移"这类隐性大坑。
