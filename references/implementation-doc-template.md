# 实现文档 + 测试设计 模板

> Step 3 Writer 起草 `impl.md` 用。要求：**具体到 AI 可按步执行**，且**每步自带测试设计**。

## impl.md 格式

```markdown
# 实现文档: <功能名称>

## 0. 依据
- PRD: .vibe-coding/<功能名称>/prd.md
- Context: .vibe-coding/<功能名称>/context.md

## 1. 数据层
- 新增/修改的表、字段、索引
- 迁移脚本/步骤
- 旧数据兼容处理

## 2. 后端逻辑
- 新增/修改的 service、domain rule、校验
- 复用的现有函数（带路径）

## 3. API 设计
- 端点、方法、鉴权
- 请求/响应结构、错误码
- 复用或扩展现有端点

## 4. 前端
- 页面/组件、UI 大致样子
- 状态管理、loading/error/success 态
- 调用哪个 API

## 5. 权限与边界
- 越权处理、输入校验、并发、重复提交

## 6. 向后兼容
- 对现有逻辑/数据/接口的影响与兼容方案

## 7. 实现步骤（细粒度，每步带测试 + 提交点）
### Step 7.1 <名称>
- 做什么：<…>
- 涉及文件：<…>
- 测试：
  - 单测: <用例>
  - 集成: <用例>
  - E2E: <用例>
- 覆盖矩阵: 正常✓ 异常✓ 权限✓ 空数据✓ 重复✓ 边界✓ 旧数据✓ 回归✓
- 验证命令: <cmd>
- 提交信息: `<type>(<scope>): <desc>`

### Step 7.2 <名称>
- …

## 8. 回归清单
- 本改动可能影响的既有功能 → 对应测试
```

## 步骤拆分要点

- **拆到可执行粒度**：不要写"实现用户邀请功能"，要拆成
  `新增字段 → 后端接口 → 业务校验 → 单测 → 前端表单 → 接入 API → 加 loading/error/success → E2E → 回归`。
- **每步独立可提交**：一步一个 commit，便于审查/回滚/定位 CI 失败。
- **测试与步骤一一对应**：没有测试的步骤不允许存在。

## 测试覆盖矩阵（每步必填）

每个关键步骤至少覆盖：

| 维度 | 说明 |
|------|------|
| 正常路径 | 主流程能跑通 |
| 异常路径 | 错误输入/失败分支 |
| 权限 | 越权被拒 |
| 空数据 | 空集合/空值 |
| 重复提交 | 幂等性 |
| 边界值 | 0/极大/极小/超长 |
| 旧数据兼容 | 历史数据不报错 |
| 回归 | 相关既有功能不破 |

> 起草后派 Reviewer 子代理审实现方案 + 测试覆盖完整性，循环至无阻断项。
