# 审查视角（Reviewer Lenses）

> vibe-coding-workflow Step 2/3 的 Reviewer 用。把"一个通用挑刺 agent"升级为**多视角 + 覆盖检查**：Reviewer 须逐个视角过一遍、打印覆盖表，全部视角无阻断项才可放行——防止提前收敛。
>
> 借鉴 autoresearch 的红队多视角审查（security-workflow 的多 adversary lens）。

## 各阶段适用的视角

| 阶段 | 适用视角 |
|:--|:--|
| **PRD（Step 2）** | 视角 1 / 2 / 3 / 5（需求范围、边界数据、安全权限、向后兼容） |
| **实现文档（Step 3）** | 5 个视角全适用，**视角 4（测试覆盖）为重点** |

## 五个视角

### 视角 1：需求·范围
- 解决的是真问题吗？和用户场景 / PRD 对得上吗？
- In/Out Scope 边界清晰？有没有范围蔓延？
- 成功标准是否**可验收**（可观测、可量化）？
- 有没有过度设计（为不会发生的场景复杂化）？

### 视角 2：边界·数据
- 边界情况是否齐全：空数据 / 重复 / 并发 / 越权 / 失败 / 超时 / 超长输入 / 旧数据？
- 数据结构、表/字段、迁移设计是否合理？
- 旧数据兼容有没有考虑？

### 视角 3：安全·权限（STRIDE 思路）
- 越权访问（水平 / 垂直）、注入、数据暴露？
- 鉴权 / 输入校验 / 敏感数据是否到位？
- 权限模型有没有漏洞？

### 视角 4：测试覆盖
- 每个实现步骤是否都带测试？
- 覆盖矩阵是否漏维度：正常 / 异常 / 权限 / 空数据 / 重复提交 / 边界值 / 旧数据兼容 / 回归？
- bugfix 的测试是否真能抓到该 bug（对照 Step 6 双分支验证）？

### 视角 5：向后兼容·回归
- 是否破坏现有逻辑 / 数据 / 接口？
- 复用：明明有现成实现却要重造？（对照 context.md）
- 回归风险点有没有列进回归清单？

## 覆盖表（Reviewer 每轮必须打印）

```
=== 审查覆盖 - <PRD | 实现文档> 轮次 N ===
视角1 需求·范围    : ✓ 无阻断 / ⚠ N 项阻断
视角2 边界·数据    : ✓ / ⚠
视角3 安全·权限    : ✓ / ⚠
视角4 测试覆盖     : ✓ / ⚠ / —（PRD 阶段不适用）
视角5 向后兼容·回归: ✓ / ⚠
--------------------------------------
阻断项合计: X ｜ 可放行: 否 / 是
```

**规则**：只有"该阶段适用的视角全部审过 + 无阻断项"才能判定可放行。任何适用视角没审 → 视为未完成，继续审。

## 用法
- 一次派一个 `general-purpose` 子代理集中审 1–2 个视角，或多视角合并审——关键是**覆盖表必须打满**。
- 每条阻断项 / 建议必须带**位置 + 证据**（见 `reviewer-prompt.md` 的"证据强制"）。
