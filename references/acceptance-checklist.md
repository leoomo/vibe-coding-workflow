# 验收门禁清单

> Step 6 用。代码写完 ≠ 完成。未过本门禁，不得进入总结。

## 0. 前置：绿基线（在 Step 5 实现前就要确认）

> 借鉴 autoresearch 的 baseline：实现前先确认仓库当前是绿的，否则后面所有"通过 / 失败"都不可信。

在开始写**任何**实现代码之前，先跑一次既有测试套件：

- **全绿** → 记为基线，开始 Step 5。
- **有红** → **停下，报告**，不开始实现。先和用户确认：修既有红，还是接受为已知红。

## 1. Guard 门禁（全量测试）

> 两个概念务必分清：
> - **verify**（Step 5 每步）：本步目标是否达成。
> - **guard**（本步）：有没有**连带破坏既有**。

跑**全部**检查，**全绿**才算过（且须**和绿基线一致或更好**）：

| 检查 | 命令（示例） | 结果 |
|------|------|------|
| 单元测试 | `npm test` / `pytest` / … | ✅/❌ |
| 集成测试 | `<cmd>` | ✅/❌ |
| E2E 测试 | `<cmd>` | ✅/❌ |
| Lint | `npm run lint` / `ruff check` / … | ✅/❌ |
| Type check | `tsc --noEmit` / `mypy` / … | ✅/❌ |
| Build | `npm run build` / `cargo build` / … | ✅/❌ |

> 命令以项目实际为准（见 context.md「测试方式 / 构建与部署」）。

## 2. Bugfix 双分支验证（仅 bugfix 任务必做）

目的：证明测试**真的覆盖了该 bug**，而非"碰巧通过"。

```
1. 切到修复前版本（修复该 bug 的 commit 的父提交）：
   - 跑相关测试 → 必须【失败】（确认测试能抓到该 bug）
2. 切到修复后版本：
   - 跑同一批测试 → 必须【通过】
```

判定：
- 修复前失败 + 修复后通过 → ✅ 测试有效
- 修复前就通过 → ❌ 测试没覆盖该 bug，补测试
- 修复后仍失败 → ❌ 修复无效，回 Step 5

## 3. 回归检查

- 跑与本改动**相关**的既有功能测试 → 仍全绿（与基线一致）。
- 参照 `impl.md` 第 8 节「回归清单」逐项确认。

## 4. 冒烟测试（可选）

- 启动应用，手动点一遍核心路径，确认主流程可用。

## 门禁结论格式

```
═══════════════════════════════════════
       🚦 验收门禁 - <功能名称>
═══════════════════════════════════════
绿基线（实现前）: ✅ / ❌（未通过则不应开始实现）

回归检查        : ✅ / ❌
Bugfix 双分支   : ✅ / ❌ / N/A
结论: 通过 → 进总结 ｜ 未通过 → 回 Step 5 修复
═══════════════════════════════════════
```
