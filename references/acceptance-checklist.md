# 验收门禁清单

> Step 6 用。代码写完 ≠ 完成。未过本门禁，不得进入总结。

## 1. 全量测试门禁

跑**全部**检查，**全绿**才算过。逐项记录结果：

| 检查 | 命令（示例） | 结果 |
|------|------|------|
| 单元测试 | `npm test` / `pytest` / … | ✅/❌ |
| 集成测试 | `<cmd>` | ✅/❌ |
| E2E 测试 | `<cmd>` | ✅/❌ |
| Lint | `npm run lint` / `ruff check` / … | ✅/❌ |
| Type check | `tsc --noEmit` / `mypy` / … | ✅/❌ |
| Build | `npm run build` / `cargo build` / … | ✅/❌ |

> 命令以项目实际为准（见 context.md「测试方式 / 构建与部署」）。

## 2. Bugfix 双分支验证（仅 bugfix 任务必做）

目的：证明测试**真的覆盖了该 bug**，而非"碰巧通过"。

```
1. 切到修复前版本（修复该 bug 的 commit 的父提交）：
   - 跑相关测试 → 必须【失败】（确认测试能抓到该 bug）
2. 切到修复后版本：
   - 跑同一批测试 → 必须【通过】
```

判定：
- 修复前失败 + 修复后通过 → ✅ 测试有效
- 修复前就通过 → ❌ 测试没覆盖该 bug，补测试
- 修复后仍失败 → ❌ 修复无效，回 Step 5

## 3. 回归检查

- 跑与本改动**相关**的既有功能测试 → 仍全绿。
- 参照 `impl.md` 第 8 节「回归清单」逐项确认。

## 4. 冒烟测试（可选）

- 启动应用，手动点一遍核心路径，确认主流程可用。
