# Context Engineering 清单

> 给 Reviewer / Writer 在 Step 1 收集上下文时用。目标：让 AI"看懂项目"，而不是凭空发挥。**优先复用既有实现，不要重复造轮子。**

## 收集清单

按以下维度用 Explore agent 深读项目，能填就填，**没有的标 N/A**：

### 1. 项目结构
- 顶层目录布局、入口文件
- 相关功能/模块所在位置（精确到文件路径）
- 框架与运行时（语言、版本、主框架）

### 2. 代码风格与约定
- 命名规范、目录组织习惯
- lint / formatter 配置（eslint/prettier/ruff/…）
- 注释密度、错误处理惯用法

### 3. 数据层
- DB schema / ORM 模型 / 迁移机制
- 是否需要新增字段、新表、数据迁移
- 现有相关表结构与关系

### 4. API 设计
- 既有相关端点（路径、方法、鉴权、版本约定）
- 请求/响应格式、错误码约定
- 是否有现成接口可复用/扩展

### 5. 业务逻辑
- 相关领域规则、核心模块与函数（带文件路径）
- 谁调用谁（调用关系）
- 本任务会触碰哪些既有逻辑

### 6. 构建与部署
- 构建/启动命令、环境变量
- CI 流水线跑了哪些检查（影响 Step 6 验收门禁）
- 部署相关约束（如不能破坏线上兼容）

### 7. 测试方式
- 测试框架、测试文件放在哪、怎么跑
- 单测 / 集成 / E2E 各自的命令
- 既有相关测试（作为风格参照与回归基础）

### 8. 可复用的现有实现
- **重点**：列出可直接复用的工具函数、组件、服务、中间件，附路径
- 已有的类似功能可作模板参照

## context.md 输出格式

```markdown
# Context: <功能名称>

## 任务类型
feature / bugfix

## 项目结构
- <路径>: <作用>

## 代码风格
- <约定>

## 数据层
- <相关表/模型/迁移机制>

## API 设计
- <相关端点>

## 业务逻辑
- <相关模块，带路径>

## 构建与部署

## 测试方式
- 单测: <cmd>｜集成: <cmd>｜E2E: <cmd>

## 可复用的现有实现
- <路径>: <可复用点>

## 风险点（初判）
- <…>
```
