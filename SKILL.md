# Vibe Coding Workflow

> 真正的 vibe coding 不是"给 AI 一句话等它写完"，而是靠一套流程，让 AI 产出**可控、可审查、可验证**。

核心铁律：**在被审查的书面计划获批准前，绝不写代码。**

人不必逐行看代码，但必须把注意力放在**流程设计、上下文质量、验收标准、风险控制**上。本 skill 把这套纪律固化成一条管线。

## 用法

```bash
# 交互模式（推荐）
/vibe-coding-workflow <任务描述>

# 指定功能名称
/vibe-coding-workflow <功能名称> <任务描述>
```

| 参数 | 说明 |
|------|------|
| `<功能名称>` | 可选，功能目录名称（kebab-case） |
| `<任务描述>` | 必需，要实现的功能或要修的 bug |

## 核心理念

AI 写出来的东西看似逻辑完整，却常藏隐患：需求理解偏差、过度设计、边界遗漏、数据结构不合理、测试不足、不顾线上兼容。本 skill 用三件事对抗这些：

1. **每份关键产出都经"对抗式 Reviewer 子代理"审查**（Writer 推进，Reviewer 挑刺，循环至收敛）。
2. **测试与实现同步设计**，并在写代码前先有被批准的规格。
3. **设计决策获批准后，全程自动**：从建分支、分步实现、逐步自动提交，到验收门禁，AI 自主完成——人只在"设计批准"这一处介入一次。

> 单个 AI 会自洽；让第二个 AI 专门找问题，是这套流程的命门。
> "批准后自动跑完"正是 vibe coding 的生产力所在：人的注意力放在设计/上下文/验收，执行交给流程。

## 文件结构

每个任务在**目标项目**内产出：

```
<项目>/.vibe-coding/<功能名称>/
├── context.md      # 上下文工程产出
├── prd.md          # 做什么（Writer+Reviewer 打磨）
├── impl.md         # 怎么做（实现文档 + 测试设计）
├── review-log.md   # Reviewer 挑刺与处置记录
└── todo.md         # 细粒度任务（每条带测试 + 提交点）
```

## 工作流程

### Step 0: 启动、命名与建分支

1. 解析输入：只有任务描述 → 派生一个功能名（kebab-case）并告知用户；给了 `<功能名> <任务>` → 直接用。
2. 判断任务类型：**feature**（新功能）还是 **bugfix**（修 bug）——bugfix 在 Step 6 有额外验证。
3. 创建 `.vibe-coding/<功能名称>/`。
4. **一开始就建独立分支**，所有后续工作（含规格文档与代码）都在此分支上：
   - feature → `git checkout -b feat/<功能名称>`
   - bugfix → `git checkout -b fix/<功能名称>`
   - 非 git 项目 → 跳过建分支，仅在总结时提示。

*WHY*：独立分支让小步自动提交可回滚、不污染主干——这是后面"批准后全自动"的前提。

### Step 1: 上下文工程（Context Engineering）

1. 用 Agent 工具的 **Explore agent**，在**项目目录内**深读相关代码。
2. 按 `references/context-checklist.md` 的清单收集：项目结构、代码风格、DB schema、API 设计、业务逻辑、构建/部署、测试方式、可复用的现有实现。
3. 把结果写入 `context.md`。

*WHY*：上游理解一旦跑偏，后面全偏。在项目内分析，AI 才能"看懂项目"而不是凭空发挥。优先复用既有实现，不要重复造轮子。

### Step 2: PRD（Writer + Reviewer）

1. **Writer**（你自己）基于 context.md 起草 `prd.md`，套用 `references/prd-template.md`：要解决什么问题、用户是谁、核心场景、范围内、范围外、边界情况、成功标准、是否涉及权限/计费/通知/数据迁移。
2. 派 **Reviewer 子代理**：用 Agent 工具派一个 `general-purpose` 子代理，把 `references/reviewer-prompt.md` 的提示词 + `prd.md` 内容一起传给它，让它**对抗式挑刺**。
3. Writer 据挑刺修订 → 再派 Reviewer → 循环至 Reviewer 报告"无阻断项"。
4. 每轮挑刺与处置记入 `review-log.md`。

*WHY*：先钉死"做什么"，再去设计"怎么做"，避免 AI 过早优化一个可能跑偏的方案。

### Step 3: 实现文档 + 测试设计（Writer + Reviewer）

1. **Writer** 基于 prd.md 起草 `impl.md`，套用 `references/implementation-doc-template.md`：
   - 数据层（表/字段/迁移）、后端逻辑、API、前端/UI、状态、错误提示、权限与边界、向后兼容。
   - **把步骤拆到可执行粒度**（如：新增字段 → 接口 → 校验 → 单测 → 前端 → 接入 → 状态 → E2E → 回归）。
   - **每步声明其测试**：单测/集成/E2E，覆盖正常/异常/权限/空数据/重复提交/边界值/旧数据兼容/回归。
2. 派 **Reviewer 子代理**审查实现方案 + 测试覆盖（同样喂 `references/reviewer-prompt.md`）。
3. 循环修订至无阻断项，记入 `review-log.md`。

*WHY*：步骤拆得越细，执行越稳、越不易跑偏；测试在文档阶段设计而非事后补，才能与代码同步产出。

### Step 4: 计划批准门禁（唯一的人工门禁）

把 PRD + impl 汇总，按下方格式呈现给用户审核。**未获批准前不得写任何代码。**

```
═══════════════════════════════════════
       📋 规格审阅 - <功能名称>（类型: feature/bugfix）
═══════════════════════════════════════

## 🎯 目标
<一句话>

## ✅ PRD 要点
- 用户/场景：<…>
- 范围内：<…>｜范围外：<…>
- 成功标准：<…>

## 🛠 实现步骤（每步带测试 + 提交）
1. <步骤> → 测试: <…>
2. <步骤> → 测试: <…>

## ⚠️ 风险与边界
- <…> → <应对>

## 📁 文件
- context.md / prd.md / impl.md / review-log.md / todo.md

═══════════════════════════════════════
回复：`确认`/`y` 开始实现 ｜ `修改`/`n` 调整 ｜ `取消`
═══════════════════════════════════════
```

> 这是整个流程**唯一**需要人介入的点。一旦确认，Step 5–7 全自动执行。

### Step 5: 分步实现 + 逐步自动提交

生成 `todo.md`（每条任务带验证命令 + 提交点），然后**一步一执行、自动提交**：

```
对 todo 里每个步骤：
  1. 实现该步代码
  2. 写/调整该步对应的测试（impl.md 已设计好）
  3. 跑该步验证（typecheck / 对应测试）
     - 失败 → 在重试上限内修复 → 重跑
     - 超过上限 → 停下，报告
  4. 通过 → git add + commit（约定式提交信息，如 `feat(invite): add invitation table`）
  5. 勾掉 todo.md，进入下一步
```

*WHY*：每个 commit = 一个可 Review / 可回滚 / 可定位 CI 失败的单元。禁止"一次性改几十个文件再提交"——看似快，风险极高。

> 这是全自动环节：设计已获批准，实现/测试/提交均自主完成，**不再逐步询问**。所有提交都在 Step 0 建的独立分支上。

### Step 6: 验收门禁（Acceptance Gate）

代码写完不等于完成。按 `references/acceptance-checklist.md` 走门禁：

1. **跑全量测试**：单测 + 集成 + E2E + lint + typecheck + build。全绿才算过。
2. **bugfix 双分支验证**（仅 bugfix 任务）：在修复前分支上确认相关测试**确实失败**，在修复后分支上确认**确实通过**——证明测试真覆盖了该 bug，而非"碰巧通过"。
3. **回归检查**：相关既有功能测试仍通过。

未过门禁 → 回 Step 5 修复，不得跳到总结。

*WHY*：逐任务绿 ≠ 整体绿；bugfix 双分支验证排除"假阳性修复"。

### Step 7: 总结与记忆确认

```
═══════════════════════════════════════
       📊 任务总结 - <功能名称>
═══════════════════════════════════════

## ✅ 完成情况
- 任务: X/Y ｜ 类型: feature/bugfix ｜ 分支: feat|fix/<名称> ｜ 提交数: N

## 💡 经验教训 / 🔄 可复用模式 / 📝 下次改进
- <…>
═══════════════════════════════════════
是否写入项目记忆？ `y` 写入 ｜ 回车/其他 不写入
═══════════════════════════════════════
```

> 不含合并 / PR 阶段。合并交由用户或其它流程决定。

## 规则（刚性，不可跳过）

- **先规格后代码**：批准的 `impl.md` 之前不写任何业务代码。
- **批准后全自动**：Step 4 获批后，Step 5–7 自主执行，不再逐步询问人。
- **独立分支**：Step 0 一开始就建分支；所有提交在该分支上，保证可回滚、不污染主干。
- **Reviewer 不可跳过**：PRD、impl 阶段必须经对抗式 Reviewer 子代理审查并收敛。
- **每步带测试 + 自动提交**：不允许"裸代码"步骤，不允许一次性大提交。
- **验收门禁强制**：全量测试须全绿；bugfix 须双分支验证。
- **复用优先**：先在 context.md 里找既有实现，再决定新增。
- **测试命令以项目为准**：验收门禁跑哪些命令，按项目实际（见 context.md「测试方式」）。

## 执行指令

当用户调用此技能时：

1. **解析输入** → 功能名 + 任务描述 + 类型（feature/bugfix）。
2. **建目录 + 建分支**：`.vibe-coding/<功能名称>/` + `git checkout -b feat|fix/<功能名称>`。
3. **Step 1**：派 Explore agent → 写 `context.md`。
4. **Step 2**：Writer 起草 `prd.md` → 派 Reviewer 子代理 → 循环修订 → 记 `review-log.md`。
5. **Step 3**：Writer 起草 `impl.md`（含测试设计）→ 派 Reviewer 子代理 → 循环修订 → 记 `review-log.md`。
6. **Step 4**：呈现规格审阅 → 等用户批准（**唯一人工门禁**）。
7. **Step 5–7 自动执行**（不再询问）：生成 `todo.md` → 逐步实现 + 测试 + 自动提交 → 全量测试门禁（bugfix 加双分支验证）→ 总结 + 记忆确认。

每一步用到的模板/提示词在 `references/` 下，按需读取，不要一次性全塞进上下文。
